1. Infrastructure 层回答三个问题

前面两课介绍了 Pipeline（做内容）和 Hugo（做页面）。Infrastructure 层解决的是怎么让这一切自动运转的问题：

问题答案
谁每天定时启动 Pipeline？GitHub Actions（定时器）
谁把生成的 .md 提交到仓库？Git（Pipeline 自己 commit + push）
谁把 Hugo 构建的 HTML 放到网上？Cloudflare Pages（自动部署）
🔑 Infrastructure = 无人值守自动化

Infrastructure 层的终极目标：你不需要每天登录服务器、不需要手动跑命令、不需要记"今天发早报了没"——系统自己完成一切。

2. 两条工作流

项目中有两个 GitHub Actions 工作流文件，它们分别做不同的事：

工作流 1：daily-digest.yml 工作流 2：deploy-blog.yml ┌──────────────────────────┐ ┌──────────────────────────┐ │ 触发：定时（每天 06:00） │ │ 触发：git push 到 main │ │ │ │ │ │ 1. 签出代码 │ │ 1. 签出代码 │ │ 2. 装 Python 依赖 │ │ 2. 装 Hugo 和 Node │ │ 3. 跑 Pipeline │ │ 3. 跑 hugo --gc --minify │ │ 4. git commit + push │ │ 4. 部署到 Cloudflare │ │ 生成的 .md │ │ Pages │ └──────────┬───────────────┘ └──────────────────────────┘ │ │ │ git push（新增 .md 文件） │ └──────────────────►────────────────────┘ 串联关系 daily-digest → deploy-blog
🔗 两个工作流是串联的

它们不是同时跑的。时序是：① 06:00 daily-digest 跑完 → ② 它 commit .md 文件并 push → ③ push 事件触发 deploy-blog → ④ Hugo 构建 → ⑤ Cloudflare 部署上线。

3. GitHub Actions 是什么

GitHub Actions 是 GitHub 自带的自动化服务。你可以把它理解成一台免费的云电脑，每次触发时它会：

  1. 给你一台临时的 Ubuntu 服务器
  2. 按你写的步骤执行命令
  3. 执行完自动销毁

你不需要自己买服务器、不需要维护操作系统——GitHub 全包了。免费版每个月有 2000 分钟的额度，对我们这个项目绰绰有余（每天跑一次，每次 3-5 分钟，月消耗不到 150 分钟）。

工作流文件的骨架

打开 .github/workflows/daily-digest.yml，它的结构是：

name: Daily AI Digest          ← 工作流名称（在 GitHub 页面上显示）

on:                            ← 触发条件
  schedule:                    ← 定时触发
    - cron: '0 22 * * *'      ← UTC 时间 22:00 = 北京时间 06:00
  workflow_dispatch:           ← 也支持手动触发

jobs:                          ← 要干的事
  digest:                      ← 任务名
    runs-on: ubuntu-latest     ← 在 Ubuntu 上跑
    steps:                     ← 步骤列表
      - uses: actions/checkout@v4    ← 步骤1：签出你的代码
      - name: Setup Python           ← 步骤2：装 Python
      - name: Run pipeline           ← 步骤3：跑管道
        env:                         ← 设置环境变量
          API_KEY: ${{ secrets.XXX }} ← 从 GitHub Secrets 读取
      - name: Commit articles        ← 步骤4：提交 .md 文件
💡 workflow_dispatch 有什么用

这是一个重要的调试功能。加上 workflow_dispatch 后，你可以去 GitHub 仓库的 Actions 页面手动点"Run workflow"按钮——这样即使没到定时时间，也能立刻触发一次运行。测试管道改动时特别好用。

4. Secrete：API Key 放在哪里

Pipeline 需要调用 DeepSeek API（需要一个密钥）。你不能把密钥直接写在代码里（任何人都能看到），而是存在 GitHub 的 Secrets（密秘） 中：

GitHub 仓库 → Settings → Secrets and variables → Actions │ ▼ ┌──────────────────────┐ │ DEEPSEEK_API_KEY │ ← 你在 GitHub 网页上设 │ CF_API_TOKEN │ ← Cloudflare 的 API 令牌 │ CF_ACCOUNT_ID │ ← Cloudflare 账号 ID └──────────────────────┘ │ ▼ 在工作流中通过语法引用： ${{ secrets.DEEPSEEK_API_KEY }} ${{ secrets.CF_API_TOKEN }}

在本地开发时，密钥存在 pipeline/.env 文件里（已加入 .gitignore，不会提交到 GitHub）。

5. Cron 定时表达式

你会在 daily-digest.yml 里看到 cron: '0 22 * * *'。这是 Linux 定时任务的格式：

┌───── 分钟 (0-59)
│ ┌───── 小时 (0-23)
│ │ ┌───── 日期 (1-31)
│ │ │ ┌───── 月份 (1-12)
│ │ │ │ ┌───── 星期 (0-7, 0=星期日)
│ │ │ │ │
0 22 * * *     ← 每天 UTC 22:00（北京时间 06:00）

一些常用例子：
0 6 * * *      ← 每天 UTC 6:00 = 北京时间 14:00
*/30 * * * *   ← 每 30 分钟
0 9 * * 1-5    ← 工作日早上 9 点
⚠️ 时区陷阱

GitHub Actions 的 cron 使用 UTC 时间（世界标准时间）。北京时间 = UTC + 8 小时。所以"北京时间每天早上 6 点"要设成 0 22 * * *（22+8=6 第二天）。

6. Cloudflare Pages：网站的"家"

Cloudflare Pages 是一个静态站点托管服务。它从你的 GitHub 仓库拉代码，自动构建 Hugo，然后把生成的 HTML 部署到全球 CDN 节点上。

工作方式：

你 git push 到 GitHub │ ▼ Cloudflare Pages 检测到变更 │ ▼ 拉取最新代码 → 执行 hugo --gc --minify │ ▼ 把 public/ 目录的内容同步到 CDN │ ▼ 全球节点生效，你的网站更新

免费版特性：

  • 无限带宽——不像 Vercel 免费版有带宽限制
  • 自动 HTTPS——免费 TLS 证书
  • CDN 加速——全球 200+ 节点
  • 每月 500 次构建——我们一天一次，绰绰有余

7. 数据流全景：从 06:00 到你在浏览器看到

把所有环节串起来，完整的时序是这样的：

06:00 北京 GitHub Actions 启动 daily-digest │ ▼ 06:01 Pipeline 开始采集新闻（RSS + API） │ ▼ 06:05 Pipeline 完成摘要，生成 8 个 .md 文件 │ ▼ 06:06 Pipeline 自动 git commit + git push │ ▼ 06:06 GitHub 检测到新 commit，触发 deploy-blog │ ▼ 06:07 Cloudflare 拉代码 → hugo 构建 → 部署 │ ▼ 06:08 ☕ 你打开手机 → 看到今天的早报

从 Pipeline 启动到网站更新，全程大约 8 分钟，不需要任何人工操作。

8. 部署博客有两种方式

方式触发方式适用场景
自动部署 你 git push 到 main 分支 平时改代码、改模板、改样式
手动触发（workflow_dispatch） 去 GitHub Actions 页面点按钮 调试部署、紧急修复、测试配置变更
