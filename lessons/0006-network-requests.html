1. 从公网获取数据的三种方式

我们的 Pipeline 有三种采集器，对应三种不同的数据获取方式。你可以把它们理解成"三种敲门方式"：

RSS 采集API 调用网页爬虫
本质读一个固定格式的订阅文件调别人写好的接口解析 HTML 网页
数据格式XML（标准化的）JSON（结构化）HTML（非结构化）
依赖对方提供 RSS feed对方提供 API 文档什么都不需要
稳定性高高低
实现难度低中高
项目对应rss_collector.pyapi_collector.pyscraper.py

一句话区别：RSS 是别人主动"喂"给你，API 是你去"问"他要，爬虫是你自己去"拿"。

2. HTTP 请求基础（三种方式都离不开）

不管哪种方式，底层都是 HTTP 请求——你的程序向远程服务器发一条消息，服务器回复一条消息。最基本的：

你的程序 目标服务器 ┌──────────┐ ┌──────────────┐ │ │ GET /api/xxx │ │ │ │ ───────────────► │ │ │ │ │ │ │ │ HTTP/1.1 200 OK │ │ │ │ ◄─────────────── │ │ │ │ {"data": ...} │ │ └──────────┘ └──────────────┘

2.1 请求方法（Method）

方法用途在我们的代码中
GET获取数据所有采集器都是 GET——只读不写
POST提交数据DeepSeek API 调用（发 prompt 拿结果）

2.2 状态码（Status Code）

服务器回复时带一个三位数的状态码，告诉我们结果：

状态码含义如何处理
200成功正常解析数据
301 / 302重定向跟随新地址（httpx 默认会自动跟）
403禁止访问（反爬）需要加 User-Agent 或换策略
429请求太频繁（限流）等一会再请求（加 sleep）
500服务器出错重试几次

在我们的代码里，resp.raise_for_status() 的作用就是：如果状态码不是 200 系列，直接抛出异常。

2.3 Headers（请求头）

HTTP 请求可以带一些"元信息"。最关键的是 User-Agent——告诉服务器"我是谁"：

# 反面教材：不设 User-Agent（服务器一眼识别是爬虫）
async with httpx.AsyncClient() as client:
    resp = await client.get(url)    # ❌ 容易被封

# 正确模仿：伪装成浏览器
headers = {
    "User-Agent": "Mozilla/5.0 (Windows NT 10.0; Win64; x64) "
                  "AppleWebKit/537.36 (KHTML, like Gecko) "
                  "Chrome/120.0.0.0 Safari/537.36"
}
async with httpx.AsyncClient(headers=headers) as client:
    resp = await client.get(url)    # ✅ 像真人浏览器

这就是 scraper.py 里 self.headers 存在的意义——我们的爬虫伪装成了一个 Chrome 浏览器。

3. 方式一：RSS 采集（最简单的）

RSS（Really Simple Syndication）是一种古老的标准化格式。网站维护一个 XML 文件，列出了最新文章的标题、链接、摘要。

3.1 代码套路

import feedparser

feed = feedparser.parse("https://example.com/rss.xml")
for entry in feed.entries:
    title = entry.get("title")
    link  = entry.get("link")
    # 正文可能在 entry.content 或 entry.summary 里

整个采集就两步：parse → 遍历 entries。不需要处理 JSON、不需要解析 HTML、不需要关心反爬。因为它本来就是给人读的。

3.2 优点和缺点

优点缺点
代码极简，库帮你搞定一切不是每个网站都有 RSS
结构化数据（标题/日期/正文都分好了）RSS 可能只提供摘要而非全文
不存在反爬问题（RSS 就是设计给程序读的）部分网站 RSS 更新不及时

3.3 本项目应用

在 rss_collector.py 中，每个新闻源都只需要在 config.yaml 里配一个 RSS 地址，不需要多写一行代码。目前配置了 openai-blog、google-ai、deepmind、huggingface、techcrunch-ai 等 9 个 RSS 源。

4. 方式二：API 调用（最规范的）

API 是网站主动提供的程序接口。你按它的规则发请求，它返回结构化的 JSON 数据。

4.1 代码套路

import httpx

async with httpx.AsyncClient() as client:
    resp = await client.get(
        "https://api.example.com/data",
        headers={"Authorization": "Bearer YOUR_TOKEN"},
        params={"page": 1, "limit": 20}
    )
    resp.raise_for_status()      # 检查状态码
    data = resp.json()            # 自动解析 JSON
    for item in data["items"]:
        print(item["title"])

GET 请求：client.get(url)
带查询参数：用 params= 字典
带请求头：用 headers= 字典
解析 JSON：resp.json()

💡 同步 vs 异步（sync vs async）

你会注意到 RSS 采集是同步的（没有 async），而 API 和爬虫是异步的（async/await）。原因：feedparser 库没有 async 版本，所以 RSS 采集是普通函数。httpx 同时支持同步和异步——我们的代码选择了异步，这样在等待网络响应时，Python 可以去做别的事。两者效果一样，只是写法不同。

4.2 本项目应用：HackerNews

api_collector.py 中的 HackerNews 采集展示了"两步走"模式：

# 第 1 步：获取热门文章 ID 列表
resp = await client.get("https://hacker-news.firebaseio.com/v0/topstories.json")
story_ids = resp.json()[:30]

# 第 2 步：逐个获取每篇文章的详情
for sid in story_ids:

5. 方式三：网页爬虫（最灵活的）

当网站没有 RSS，也没有 API，就只能爬虫——直接下载 HTML 页面，然后从 HTML 中提取信息。

5.1 代码套路

import httpx
from bs4 import BeautifulSoup

# 1. 下载网页
resp = httpx.get("https://example.com", headers=headers)
soup = BeautifulSoup(resp.text, "lxml")

# 2. 用选择器提取信息
for item in soup.select(".article-title a"):
    title = item.get_text(strip=True)   # 提取文本
    url   = item.get("href")            # 提取链接属性

关键在于第 2 步——你需要知道目标网站的 HTML 结构，才能写出正确的选择器。

5.2 CSS 选择器入门

BeautifulSoup 的 soup.select() 使用 CSS 选择器语法：

选择器含义示例
h2所有 <h2> 元素标题
.title所有 class="title" 的元素.article-title
h2 a<h2> 内部的 <a>文章链接
h2 > a<h2> 的直接子 <a>更精确
.entry-title aclass 为 entry-title 内的 <a>和项目代码一样
a[href*='/article']href 包含 "/article" 的 <a>机器之心爬虫

5.3 本项目中的爬虫

我们的 scraper.py 针对量子位和机器之心写了不同的 CSS 选择器：

量子位 
soup.select(
  "article h2 a, "
  ".entry-title a, "
  "h3 a"
)
尝试三种可能的选择器，用逗号分隔（OR 逻辑）
机器之心 
soup.select(
  "a[href*='/article']"
)
找所有链接中包含 "/article" 的 <a>

为什么两个站点的写法不同？因为每个网站的前端写的 HTML 不一样。爬虫必须"因站制宜"。

6. 爬虫的通用思维模式

写爬虫的思考步骤总是这三步：

第 1 步 第 2 步 第 3 步 分析目标 编写选择器 提取数据 ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │ 打开网页 │ │ 找到文章列表 │ │ 拿标题 │ │ F12 看结构 │ │ 找出规律 │ │ 拿链接 │ │ 找到文章区域 │ │ 写匹配规则 │ │ 拿日期 │ └─────────────┘ └─────────────┘ └─────────────┘

7. 反爬策略与对策

有些网站不希望被爬虫抓取，会设置各种反爬机制。以下是完整的对抗阶梯，从基础到高阶：

🔵 基础级

反爬类型特征描述应对策略代码示例难度成功率
User-Agent 检查 返回 403 或空数据 伪装成浏览器 UA headers["User-Agent"] = "Mozilla/5.0 (...)" 🟡 低 90%
Referer 来源校验 拦截跨站请求，提示"非法请求来源" 补充合法 Referer headers["Referer"] = "https://www.google.com/" 🟡 低 80%
IP 频率限制 高频请求返回 429，临时封禁 IP 加延时，控制并发在 3-5 个 await asyncio.sleep(2) 🟠 中 90%
静态资源防盗链 图片无法加载，显示 403 占位图 图片请求头加 Referer headers["Referer"] = "https://site.com/" 🟡 低 70%

🟠 进阶级

反爬类型特征描述应对策略代码示例难度成功率
Cloudflare 人机验证 弹出验证中间页（滑块/点选） 用 cloudscraper 自动验证 cloudscraper.create_scraper() 🟠 中高 60%
JS 动态渲染 原始 HTML 不含数据 Playwright 等 JS 执行完再取 HTML page.wait_for_selector() + page.content() 🟡 高 50%
图形/滑块验证码 频繁访问时弹出验证码 第三方打码平台 / Playwright + OCR slider.drag_to(target, steps=20) 🟡 高 40%
字体反爬 文字乱码，源码为特殊 Unicode 下载 .woff 解析字符映射 TTFont("custom.woff").getBestCmap() 🟡 高 30%
请求参数签名 接口带加密参数，错误返回 403 逆向 JS，用 Python 复现签名 hashlib.md5(...) 🟡 高 40%

🔴 高阶

反爬类型特征描述应对策略代码示例难度成功率
设备/浏览器指纹 检测 Canvas/WebGL 识别自动化浏览器 Playwright 禁用自动化特征 --disable-blink-features=AutomationControlled 🟢 极高 30%
行为轨迹检测 检测鼠标/滚动轨迹识别非人类 模拟带步骤的移动和随机停顿 page.mouse.move(100, 200, steps=10) 🟢 极高 25%
登录权限校验 未登录无法看正文 持久化登录 Cookie context.cookies() 复用会话 🟡 高 35%
蜜罐陷阱 隐藏不可见链接，爬虫访问即封禁 过滤隐藏元素 soup.select('a[style*="display:none"]') 🟠 中 20%
代理 IP 检测 通过请求头检测并封禁代理 IP 用高匿代理，移除代理请求头 headers.pop("X-Forwarded-For", None) 🟡 高 20%
🔑 反爬对抗的第一原则

先用最简单的方法，逐级升级。大多数网站只需要 User-Agent 伪装就够了。遇到反爬时从表格最上面开始试，而不是一上来就上 Playwright。

8. 为什么我们的中文源都没抓到数据？

回顾一下 pipeline 运行结果：量子位 0 篇、机器之心 0 篇。可能的原因：

  1. CSS 选择器过时了——网站在改版，之前的 class 名称变了
  2. 反爬拦截——简单伪造 User-Agent 不够了
  3. JavaScript 渲染——内容是通过 JS 动态加载的，httpx 拿不到

如果是问题 3，就需要升级到 Playwright 方案（上面表格最后一行）。我们目前没有这个必要——英文源已经提供足够的内容。

9. httpx 快速参考

httpx 是项目中使用的主力 HTTP 库。以下是最常用的操作：

import httpx

# ── 同步请求（RSS 采集器用） ──
resp = httpx.get(url)
resp = httpx.get(url, headers=headers, params=params)
resp.raise_for_status()
data = resp.json()

# ── 异步请求（API/爬虫用） ──
async with httpx.AsyncClient() as client:
    resp = await client.get(url)
    resp = await client.get(url, headers=headers)
    resp = await client.get(url, params={"key": "value"})

# ── POST 请求（DeepSeek API 用） ──
resp = await client.post(
    "https://api.deepseek.com/chat/completions",
    json={"model": "deepseek-chat", "messages": [...]}
)

# ── 常用属性 ──
resp.status_code     # 200, 403, 429
resp.text            # 返回的文本（HTML/XML）
resp.json()          # 返回的 JSON
resp.headers         # 响应头

10. 其他网络请求方式

除了 RSS、REST API、网页爬虫，还有一些其他的数据交换方式值得一提——你在其他项目或工作中可能会遇到：

方式传输方向数据格式典型场景Python 库
WebSocket 双向实时 任意（常见 JSON） 聊天、股票行情、实时通知 websockets
GraphQL 客户端查询 JSON 灵活的数据查询（GitHub API 新版） gql / httpx
gRPC 双向（支持流） Protobuf（二进制） 微服务内部通信、高性能场景 grpcio
SSE 服务端→客户端 文本流 AI 流式输出（如 DeepSeek 的 streaming） httpx / sse-starlette
Webhook 服务端→你的服务器 JSON（常见） GitHub push 通知、支付回调 Flask / FastAPI（接收端）

10.1 WebSocket

HTTP 是"一问一答"：客户端发请求，服务器回复。WebSocket 是"打电话"：建立连接后，两边随时可以说话。

在我们的项目中暂时没用到，但如果你以后想做"实时推送早报到读者浏览器"的功能，就需要 WebSocket。

import asyncio
import websockets

async def listen():
    async with websockets.connect("wss://example.com/socket") as ws:
        while True:
            msg = await ws.recv()  # 等服务器推消息
            print(f"收到: {msg}")

asyncio.run(listen())

10.2 GraphQL

REST API 的"升级版"。REST 的接口是固定的（/users 返回用户列表，/posts 返回文章列表），GraphQL 让你自己决定要什么字段：

# REST: GET /users → 返回 id, name, email, avatar, phone, address...
# 你不需要 address，但 REST 还是全给你

# GraphQL: 你需要什么自己写
query = """
query {
  users {
    id
    name
    email        # 只要这三个
  }
}
"""
resp = httpx.post("https://api.example.com/graphql", json={"query": query})

GitHub 的 API 同时提供 REST 和 GraphQL 两种方式。GraphQL 对爬虫不太友好（通常需要认证），但对前端开发者来说很灵活。

💡 REST 还是 GraphQL？

REST 像菜单上固定套餐（只能选套餐A/B/C）。GraphQL 像自助餐（自己挑菜）。REST 简单直接，GraphQL 灵活但复杂。对采集数据来说，REST 够用。

10.3 gRPC

谷歌开发的高性能 RPC 框架。数据不是 JSON 而是 Protobuf（二进制），所以比 HTTP/JSON 快很多。通常用在微服务之间，而不是浏览器和服务器之间。

你大概率不会在个人项目里遇到它，但如果你去大厂工作，内部服务通信很可能用的就是 gRPC。

# 先定义 .proto 文件
# service NewsService { rpc GetDigest(Date) returns (Digest); }

# Python 调用
response = stub.GetDigest(Date(today="2026-07-05"))
print(response.title)  # 直接拿到结构化数据

10.4 SSE (Server-Sent Events)

SSE 和 WebSocket 类似，但是单向的——服务器往客户端推数据，客户端不能往回发。实现也简单得多，就是 HTTP 长连接返回一段不断追加的文本流。

DeepSeek / OpenAI 的流式输出就是 SSE。当你看到 AI 一个字一个字地输出时，底层就是 SSE：

# 服务器返回的数据流：
data: {"content": "Open"}
data: {"content": "AI"}
data: {"content": " "}
data: {"content": "发"}
data: [DONE]

在 httpx 中读取 SSE 流：

async with httpx.AsyncClient() as client:
    async with client.stream("POST", url, json=body) as resp:
        async for line in resp.aiter_lines():
            if line.startswith("data: ") and line != "data: [DONE]":
                chunk = line[6:]
                print(chunk)  # 逐字输出

10.5 Webhook（反向 API）

通常是你去"拿"数据，Webhook 是反过来——服务器"主动"把数据发给你。

举个例子：你在 GitHub 仓库设置了一个 Webhook，URL 是你自己的服务器。之后每次有人 push 代码，GitHub 就向你的服务器发一个 POST 请求，告诉你"有人 push 了"。

这正是我们项目中 deploy-blog.yml 的触发机制——GitHub 检测到 push 事件，通知 Cloudflare Pages 开始构建。

10.6 选型建议

遇到一个新项目需要"从网上拿数据"时，按这个优先级思考：

  1. 有 RSS？ → 用 RSS（最简单）
  2. 有官方 API？ → 用 API（最稳定）
  3. 有 GraphQL？ → 用 GraphQL（如果 REST 不够灵活）
  4. 都没有？ → 写爬虫（最灵活）
  5. 需要实时推送？ → WebSocket / SSE
  6. 高性能内部通信？ → gRPC
