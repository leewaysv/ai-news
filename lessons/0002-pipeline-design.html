🏗️ Pipeline 实现思路与技术细节

Lesson 2 · 预计阅读：15 分钟 · 难度：入门 · 关联：术语表

1. Pipeline 是什么

Pipeline（管道）是项目的大脑——一个 Python 程序，负责每天早上自动完成新闻编辑的工作：

采集新闻 → 筛选过滤 → AI 摘要 → 质量检查 → 格式转换 → 发布 │ │ │ │ │ │ gather/ filter/ summarize/ qa/ adapt/ publish/

整套流程是单向流水线：前一步的输出是下一步的输入。没有回头路，没有循环——每一步只管自己的事，然后把结果传给下一步。

🔑 理解 Pipeline 的关键

这就好比一条工厂生产线。原材料（新闻）从一头进来，经过每道工序，最后变成成品（.md 文章文件）从另一头出去。你不需要理解每个工序的全部细节——但要知道它们怎么衔接。

2. 数据流：文章在管道中的"变形"

文章在流过管道的过程中，形态变了三次。理解这三步变形就理解了整个系统的一半：

步骤数据形态从哪来到哪去
① 采集（gather） RawArticle 互联网（RSS/API/爬虫） 过滤器
② 摘要（summarize） ProcessedArticle LLM 处理后 质量检查
③ 发布（publish） .md 文件 适配器转换后 Hugo 的 content/ 目录

这三个数据形态在 pipeline/src/models.py 中定义，用的 Pydantic 库。你可以打开看看——每个字段都有中文注释，很好懂。

3. 六步拆解

Step 1: gather/（采集）

三种采集方式对应三种新闻源类型：

采集方式对应源Python 文件
RSSOpenAI Blog、Google AI 等有 RSS 的站点rss_collector.py
APIHackerNews、GitHub Trendingapi_collector.py
Scrape（爬虫）机器之心、量子位（无 RSS）scraper.py

采集器的返回值都是 list[RawArticle]。主程序把所有采集器的结果合并成一个列表。

💡 异步（async）不是必须懂的

你可能注意到 api_collector.py 和 scraper.py 用了 async/await。这是因为它们要等网络响应（HTTP 请求），Python 的 async 让这段时间可以去干别的事，而不是干等。RSS 采集器用的 feedparser 没有 async 版本，所以是同步的。混着用没毛病。

Step 2: filter/（筛选）

从几百条新闻中挑出 AI 相关的内容。分为两级，成本从低到高：

第一级：关键词过滤（免费）
用 AI_KEYWORDS 列表检查标题和正文。命中关键词越多得分越高。得分低于阈值直接扔。这一步会砍掉 60-70% 的无关内容（体育、娱乐、纯财经等）。
第二级：LLM 精判（花钱）
通过第一级的文章（约 30-40 篇），每 10 篇一批递给 DeepSeek，让它判断哪些"与 AI 行业高度相关"。这一步会进一步筛选到 10-15 篇。
💰 为什么需要两级过滤

LLM 调用是按量计费的。如果一次性把 100 篇原文都丢给 DeepSeek，不仅慢，而且贵。先用免费的关键词规则砍掉大部分噪声，只让 LLM 处理"已经有点像 AI 新闻"的候选项——这是整条管道里最重要的成本控制策略。

另外，在过滤之前还有一步 dedup（去重）。同一篇新闻可能被多个源报道（比如 OpenAI 自己的 Blog + TechCrunch 的报道），用 URL 和 hash 去重，避免重复内容。

Step 3: summarize/（摘要）

这是调 LLM 的核心环节：

  1. 单篇摘要：对每篇文章调用 DeepSeek，要求输出中文标题、1-2 句摘要、关键要点、分类标签
  2. 早报标题：全部摘要完成后，再把文章列表汇总给 DeepSeek，让它为今天的早报想一个总标题

提示词模板在 summarize/prompt.py 里，要求 LLM 返回严格 JSON 格式：

{
  "title": "中文标题",
  "summary": "中文摘要",
  "key_points": ["要点1", "要点2"],
  "categories": ["分类1"],
  "tags": ["tag1"]
}

Step 4: qa/（质量检查）

验证 LLM 生成的摘要"没乱说"。两种机制：

  • Embedding 相似度：用 sentence-transformers 把原文和摘要都转成向量，算余弦相似度。低于阈值说明摘要可能偏离原文，丢弃。
  • 兜底策略：如果当天通过 QA 的文章不足 3 篇，跳过发布，不发"没什么好说的"那种早报。
🔧 当前状态：跨语言跳过

目前咱们的源主要是英文，但摘要是中文。英文原文和中文摘要的 embedding 相似度天然偏低，所以 QA 模块对英文源跳过了校验。如果你以后加了中文源（原文是中文、摘要是中文），QA 就会实际生效。

Step 5: adapt/（适配）

把内部数据模型转为 Hugo 能识别的 Markdown 文件。具体工作：

  • 生成 YAML frontmatter（标题、日期、分类、标签等）
  • 把摘要和 key_points 排版成 Markdown 正文
  • 生成文件名（如 2026-07-04-en-openai-gpt-56-sol.md）

适配器使用策略模式——以后如果添加微信或抖音的适配器，只需要新增一个类，不影响已有代码。

Step 6: publish/（发布）

最后一步最简单：把适配器生成的字符串写入对应的文件路径。

4. 配置驱动：config.yaml 是控制中心

Pipeline 的大部分行为由 config.yaml 控制，改配置不需要改代码：

# 改这里就能控制管道行为：
llm:
  model: "deepseek-chat"      # 换模型
  max_tokens: 4096            # 控制摘要长度

pipeline:
  max_articles_per_digest: 8  # 每天最多发几篇
  min_articles_to_publish: 3  # 少于几篇就跳过
  source_weights:
    openai-blog: 10           # 热度排序权重
    default: 5

5. 主程序编排：main.py 怎么串起来

main.py 里的 run() 方法就是管道的总指挥。它的核心结构：

async def run(self):
    # 1. 采集 —— 分别从 RSS、API、爬虫收集
    all_articles = []
    for src in rss_sources:   all_articles += fetch(src)
    for src in api_sources:   all_articles += await fetch(src)

    # 2. 去重
    all_articles = dedup(all_articles)

    # 3. 过滤 —— 先关键词后 LLM
    filtered = keyword_filter(all_articles)
    filtered += await llm_filter(filtered)

    # 4. 摘要 —— 每篇调一次 LLM
    processed = [await summarize(a) for a in filtered]

    # 5. QA
    passed = qa_check(processed)

    # 6. 适配 + 发布
    adapted = blog_adapter.adapt(passed)
    count = publisher.publish(adapted)

代码的实际写法比这略复杂（有错误处理、统计日志等），但核心骨架就是这样。

6. 验证理解

如果能回答这些问题，说明你理解了 Pipeline 的核心：

  1. 为什么需要先关键词过滤再 LLM 过滤，顺序反了会发生什么？
  2. Pipeline 最终输出的是什么文件？放到哪个目录？
  3. 如果某天 DeepSeek API 不稳定、请求超时了，管道会直接崩溃吗？（提示：看 try/except）
  4. 如果我想增加一个新新闻源（比如新增一个 RSS 源），需要改 Python 代码吗？
  5. 数据模型中的 RawArticle 和 ProcessedArticle 有什么区别？
